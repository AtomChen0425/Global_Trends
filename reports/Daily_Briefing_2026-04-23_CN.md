## Firefox IndexedDB 漏洞分析报告

**背景**

近期发现了一个影响所有 Firefox 系列浏览器的隐私漏洞。该漏洞允许网站通过观察 IndexedDB API 返回条目（数据库名称）的顺序，来生成一个稳定且可复用的浏览器进程级标识符。这一发现尤其令人担忧，因为它绕过了用户在隐私浏览模式下对隔离性的预期，甚至在 Tor Browser 的“新身份”功能下也能保持其有效性。

**技术实现**

该漏洞的核心在于 `indexedDB.databases()` API 在 Firefox 中的实现。在隐私浏览模式下，该 API 返回的数据库名称顺序并非按照创建时间或字母顺序，而是基于内部存储结构（例如哈希表）的映射顺序。这种顺序是进程级别的，这意味着不同来源（Origin）的网站在同一浏览器进程运行时，可以独立观察到相同的数据库名称顺序，从而实现跨站追踪。更严重的是，在 Tor Browser 中，即使执行了“新身份”操作，该标识符也能在浏览器进程存活期间持续存在，有效抵消了“新身份”旨在提供的匿名性。

**应用场景与影响**

此漏洞打破了用户对隐私浏览和隐私保护浏览器的基本信任。用户期望不同网站之间是隔离的，并且隐私会话结束后相关状态应被清除。然而，该漏洞使得网站无需依赖 Cookie 或 LocalStorage 等传统追踪手段，仅通过观察 IndexedDB API 的返回顺序，就能构建一个高容量的跨站追踪器。这对于依赖匿名性的用户群体，如 Tor Browser 用户，构成了严重的隐私威胁。

**总结与建议**

该漏洞的根源在于暴露了浏览器内部实现细节，而非直接的数据泄露。Mozilla 已迅速发布了修复补丁（Firefox 150 和 ESR 140.10.0），其修复思路是规范化或排序 API 返回的结果，移除其中包含的进程级状态信息。此事件提醒开发者，即使是看似无害的 API，如果泄露了稳定的进程级状态，也可能成为跨站追踪的向量。对于安全和产品团队而言，应警惕任何可能暴露内部实现细节的 API，并确保其在设计上符合隐私保护的最高标准。
